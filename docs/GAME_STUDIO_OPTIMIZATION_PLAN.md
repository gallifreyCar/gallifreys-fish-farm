# Gallifrey's Fish Farm Game Studio 优化路线图

> 目标：把当前 Flutter 原型推进为“可持续扩内容、可增强游戏感、可逐步混合引擎”的挂机钓鱼游戏
> 状态：2026-03-28

---

## 1. 产品判断

当前项目最有潜力的卖点不是纯挂机，而是：

- 钓鱼收集
- 鱼宠养成
- 村庄经营
- 自动战斗
- 轻量事件驱动

这意味着它最适合走“轻策略放置 + 角色收集 + 可视化鱼村”的方向，而不是做成纯数值挂机器。

---

## 2. 当前核心问题

### 2.1 玩法层

- 玩家日常决策偏少，更多是等待数值增长
- 稀有鱼获取主要靠概率，缺少事件和目标驱动
- 建筑系统已有协同设定，但玩家感知弱
- 战斗前准备深度不足，配队价值还不强

### 2.2 架构层

- 游戏模拟、存档状态、界面渲染仍有耦合
- 世界场景仍偏 UI 原型式实现，不利于加入更强动画层
- 本地存档仍适合早期规模，不适合后续大量实体与赛季活动

### 2.3 表现层

- 世界、战斗、资源反馈大多是文字/emoji 驱动
- 视觉上已经能玩，但“钓到稀有鱼”“活动开启”“Boss 出场”的冲击感不够

---

## 3. 推荐技术路线

### 3.1 近期路线

- 保持 Flutter 作为主框架
- Riverpod 继续承担游戏状态管理
- 优先推进模型不可变化和领域拆分
- 用活动系统和成长分支提升中期留存

### 3.2 中期路线

- 存档迁移到 `Isar`
- 世界场景和战斗场景逐步迁到 `Flame`
- Flutter 保留 HUD、菜单、图鉴、商店、成就页

### 3.3 为什么不是直接重写

- 现有代码已经有玩法雏形和平台能力
- 直接重写会把时间花在重复造轮子
- 更适合的方式是：先把系统边界整理干净，再局部换实时渲染层

---

## 4. Flame + Flutter 混合架构建议

### 4.1 边界定义

- Flutter 管理：导航、面板、弹窗、商城、图鉴、成就、教程
- Flame 管理：世界场景、鱼移动、资源飘字、Boss 出场、战斗特效
- Provider 管理：玩家数据、建筑状态、队伍、活动、任务、存档

### 4.2 推荐落地顺序

1. 主世界先迁
2. 战斗场景再迁
3. 粒子特效和资源化美术最后补

## 5. 玩法升级路线

### 5.1 让玩家“回来有事做”

- 限时活动：收入、稀有鱼、Boss 奖励、装备掉落窗口
- 回归结算选择：金币补给 / 鱼食补给 / 稀有鱼雷达 三选一
- 每日钓点：不同时间段主推不同鱼群

### 5.2 让鱼从资源变成角色

- 鱼词条：勤奋、幸运、耐打、贪吃、急躁
- 鱼性格：影响工作效率和战斗风格
- 羁绊：同海域、同稀有度、同职业触发小队加成
- 专精：每条鱼走赚钱线、战斗线、探索线

### 5.3 让建筑从按钮变成策略

- 码头流：偏钓鱼速度和稀有率
- 商店流：偏出售价格和金币爆发
- 农田流：偏鱼食、成长、养成效率
- 神殿流：偏高稀有召唤与活动联动

### 5.4 让战斗前有准备意义

- Boss 拥有标签：甲壳、深海、暴怒、潮汐
- 鱼队拥有针对属性：破甲、治疗、暴击、控制
- 战斗前加入潮汐预报和推荐配队提示
